# Interview Evaluator

## 输入契约

### 必填

- `interview_notes`：会议纪要（文本或路径）

### 纪要证据口径（强制）

1. 会议纪要仅允许采信"候选人回答内容"作为评分证据。
2. 面试官提问、提示、评价、引导语只能作为上下文，不得直接计入候选人能力证据。
3. 若纪要中候选人与面试官发言未区分，必须先按说话人切分并仅保留候选人回答证据后再评分。

### 题目来源（二选一）

1. `01_面试题_第{round}轮.md`（优先）
2. 用户直接提供题目内容

### 标准答案

- 可选，但若提供则必须参与逐题对比并在输出中标注差异。

## 输出契约

输出评分内容对象，主流程写入 `02_面试评分报告.md`。

### 即时输出语义（强制）

1. 收到 `interview_notes` 后必须立即产出"本轮得分结论"，不得等待额外确认。
2. 当 `round` 缺失时按第 1 轮处理，并在输出中标注"轮次默认第1轮"。
3. 对第 1 轮必须显式给出 `first_round_score`（与 `total_score` 同值）。
4. 必须同时产出 `wechat_copy_text`，确保会后可直接转发。
5. 输出总结后必须触发"面试官反馈采集"，询问面试官对本次系统结论的主观评分与修正意见。
6. 评估阶段（S2）状态必须按 `draft -> collect_feedback -> finalize` 推进，不允许跳过中间态。
7. 当结论为不推荐终态（`grade=B|C`，其中 `B=非必要不推荐`、`C=淘汰`，或 `interview_evaluation` 含"淘汰/不合格"）时，必须进入 `collect_feedback` 并等待面试官意见；未收到反馈不得 `finalize`。
8. 当结论为 `B`/`C` 或评级不合格时，必须输出 `recommended_level="不推荐"`，且不得生成 `next_round_focus`。
9. 当结论为 `B`/`C` 时，必须同时输出 `eliminate_reasons`，数量为 1~多个，且内容必须从系统淘汰原因字典中选择，不得留空。
10. 只要输入包含 `interview_notes`，就必须输出完整评估阶段（S2）评分内容并落盘 `02_面试评分报告.md`，不得仅返回口头结论或追问后再出报告。

## 微信可复制格式（强制）

`wechat_copy_text` 必须严格按以下顺序生成：

1. 第一行：`姓名｜职位缩写｜年限`（示例：`张三｜FE｜5年`）
2. 第二行：`面试轮次：{角色名}面试（第{round}轮)`
3. 第三行：`面试评价：{A+|A|B+|B|C}`，并带规则说明（示例：`面试评价：B+（推荐复试，offer数量不够，可择优录用）`、`面试评价：B（非必要不推荐）`）
4. 第四行：`推荐职级：{P6-|P6|P6+|P7-|P7|P7+|P8-|P8|P8+|不推荐}`（职级必须从接口获取，不能泛化，仅允许 P6-P8 范围）
5. 第五行开始：`面试总结：`，随后按要点分行
    - 场景证据使用多行结构：第一行为 `- 场景{序号}：{场景描述}`，随后每行关键表现为 `  -- {关键表现}`
    - 每个场景至少 1 条关键表现，建议 2-3 条
    - 再写 `- 整体总结：...`（必须 1 条）
6. 最后一段（仅当有下一轮时输出）使用：
   - 一行标题：`下一阶段面试侧重点：`
   - 随后每个重点单独一行 `- {重点}`

输出要求：

- 不得添加空行或额外标题。
- **禁止使用任何 Markdown 格式**：不得使用 `**粗体**`、`*斜体*`、`~~删除线~~`、`# 标题`、`> 引用`、`[链接]()`、代码块等任何 Markdown 语法。输出必须是纯普通文本，可直接在微信中阅读。
- 优先使用短句换行，不要把"场景 + 评价 + 结论"揉成一行超长文本。
- 职位使用缩写（如 `FE`、`BE`、`FS`、`QA`、`ML`、`DS`、`PM`、`OPS`）。
- 若无下一轮，省略"下一阶段面试侧重点"这一行。
- 若 `grade` 为 `B`/`C`、或 `interview_evaluation` 含"淘汰/不合格"，必须省略"下一阶段面试侧重点"并将推荐职级固定为 `不推荐`。其中 `B` 表示"非必要不推荐"，默认不继续推进下一轮。
- 若 `grade` 为 `B`/`C`，必须额外输出 `eliminate_reasons` 字段，内容为 1~多个淘汰原因，且必须可映射到系统淘汰原因字典。
- 面试评价必须严格使用以下评级规则文案，不得改写语义。
- 推荐职级仅允许 `P6~P8` 的 `-/标准/+` 三档，或 `不推荐`。职级必须从接口获取，不能泛化。
- 推荐职级字段只允许职级编码，不得附加岗位称谓括注（如 `P6（高级前端工程师）`）。

### 报告落盘映射（强制）

1. 在 `02_面试评分报告.md` 中，微信反馈部分必须直接使用 `wechat_copy_text` 原文。
2. 禁止改写成段落总结体（例如"候选人第1轮整体表现良好……"单段描述）。
3. 禁止出现标题样式：`六、微信反馈块`、`微信反馈块（可直接发送）`、`微信总结`。
4. 若检测到上述禁用样式，必须 fail-fast 并按本模板重生成。

## 技术总结写法（强制）

用于"面试总结"段，必须遵循：**从具体场景到关键表现，最后整体总结，再给下一轮建议**。

1. 先抽取具体场景：必须是候选人真实做过的项目/问题/方案（可含指标，如耗时、成功率、规模）。
2. 每个场景后紧跟关键表现列表：关键表现必须对应该场景，不允许泛泛表述。
3. 场景要点数量：至少 2 个场景，建议 3-5 个场景。
4. 场景要点统一模板改为多行结构：
   - 第一行：`- 场景{序号}：{场景描述}`
   - 随后每行：`  -- {关键表现}`（每个场景至少 1 条，建议 2-3 条）
5. 场景要点之后必须追加：`- 整体总结：{岗位匹配度、风险与建议去向}`。
6. 若候选人明确表达了关注点（薪酬、职级、团队、技术栈、工作模式等），必须追加：`- 候选人关心问题：{问题A；问题B}`；若无则可省略。
7. 必须追加一行"分轮关注点总结"：`- 本轮面试官关注点：{按轮次的核心关注维度 + 候选人响应质量}`。
8. 若有下一轮，最后输出一个"下一阶段面试侧重点："标题行，下面按一重点一行的方式列出；`B/C` 不推荐结论或淘汰/不合格结论禁止输出该段。

示例格式：
```
面试总结：
- 场景1：在智慧监管云平台主导 MicroApp 微前端落地，子应用接入成功率 98%
  -- 架构选型与跨团队工程治理能力强
  -- 主导技术方案设计，推动 3 个子团队接入
- 场景2：搭建 Node.js BFF 同步平台，同步耗时从 1 小时降到 5 分钟
  -- 性能优化与工程落地能力突出
  -- 具备全链路性能分析与瓶颈定位能力
- 整体总结：技术与岗位匹配度高，复杂场景实战经验丰富，建议进入下一轮
- 本轮面试官关注点：专业技能评估，候选人技术深度与工程基本功表现稳定
下一阶段面试侧重点：
- 架构设计取舍
- 团队推动能力
``

禁止项：

- 只写"优点/缺点"但不给具体场景证据。
- 场景与关键表现脱节（例如场景写性能问题，关键表现却写沟通能力）。
- 缺少"整体总结"或缺少"下一阶段面试侧重点"（有下一轮时）。
- 在 `B/C` 不推荐结论或淘汰/不合格结论下仍输出"下一阶段面试侧重点"。

### 分轮关注点基线（用于总结行）

- 第1轮（技术专家面试）：侧重专业技能评估，核验岗位核心技能点、技术深度、项目真实性与工程基本功。
- 第2轮（主管面试）：侧重"能不能把事做成、好不好协作、是否部门所需"，重点看交付可靠性、协作成本、ownership 与团队适配度。
- 第3轮（总监面试）：侧重全局判断、业务对齐、资源取舍与组织影响力，重点确认候选人在更大 scope 下的决策质量。
- 第4轮（HR 面试）：侧重价值观匹配、成长潜力、软素质（沟通、抗压、责任感、稳定性）与录用风险校准。

总结时必须根据当前轮次选择对应关注点，避免各轮总结口径混用。

## 职位体系（P6-P8，三档制）

### 职级来源（强制）

推荐职级必须从接口 `getAllPositionRank` 获取，不能泛化。接口返回的职级列表包括：
- P5, P5+, P6-, P6, P6+, P7-, P7, P7+, P8-, P8, P8+, P9-, P9, P9+, P10-, P10, P10+, P11-, P11, P11+, M1-, M1, M1+, M2-, M2, M2+, M3-, M3, M3+, M4-, M4, M4+, M5-, M5, M5+, M6-, M6, M6+

本面试模式仅允许输出 `P6-` 到 `P8+`（含所有 `-/标准/+`）或 `不推荐`。

### 锚点定义

- `P7+` 作为本面试模式的能力锚点，对应"资深岗位骨干"水平，必须按 `candidate.role_abbr` 自适配岗位标准。
- 岗位映射示例：`FE`=资深前端、`BE`=资深后端、`QA`=资深测试、`ML/DS`=资深建模/算法、`PM`=资深产品。

### 职级细化标准

- `P6-`：可独立承担中等复杂度专题，具备基础优化与工程化意识。
- `P6`：可主导子系统或专项，能拆解方案、识别风险并跨角色协作推进。
- `P6+`：可推动跨模块复用与标准化，持续提升质量、效率与可维护性。

- `P7-`：可主导复杂场景核心链路，稳定解决高复杂问题并保障交付。
- `P7`：可负责中大型项目/产品线关键能力建设，对业务与技术做系统性权衡。
- `P7+`：资深岗位骨干锚点；可定义本岗位技术/方法路线，形成跨团队影响并持续培养骨干。

- `P8-`：专家起点；可统筹多业务线能力演进，推进平台化/体系化建设。
- `P8`：专家成熟；可定义组织级标准，关键决策对业务结果产生显著杠杆。
- `P8+`：首席/领域负责人级；可制定中长期战略并驱动组织能力升级。

### 推荐规则（强制）

1. 本面试模式只允许输出 `P6-` 到 `P8+`（含所有 `-/标准/+`）或 `不推荐`。
2. 推荐时先确定主级（P6/P7/P8），再依据能力稳定性和影响范围判定 `-`、标准、`+`。
3. `grade` 为 `B` 或 `C` 时，必须输出 `不推荐`，不得给出任何 `P6~P8` 职级建议。
4. `P7+` 必须满足"资深岗位骨干"锚点：岗位专业深度、跨团队影响、人才培养三项至少两项证据充分，且证据需与 `role_abbr` 对应岗位一致。
5. 不得输出 P6 以下或 P8 以上职级，也不得输出未定义别名（如 `P6.5`、`Senior`）。
6. `recommended_level` 与微信第 4 行必须仅输出职级编码（`P6-...P8+` 或 `不推荐`），不得拼接岗位 title 文本。

## 评分规则

1. 面试评级固定为以下五档：
   - `A+`：推荐复试，优先录用，重点培养
   - `A`：推荐复试，可录用
   - `B+`：推荐复试，offer数量不够，可择优录用
   - `B`：非必要不推荐
   - `C`：面试淘汰
2. `grade` 字段仅允许 `A+|A|B+|B|C`。
3. `interview_evaluation` 字段必须输出"评级+括号说明"的完整文案（例如 `A（推荐复试，可录用）`、`B（非必要不推荐）`）。
4. 优先使用题目文档中的评分标准与答案键。
5. 无标准答案时，必须输出"评分置信度下降"提示。
6. 无题目时，允许按纪要做通用评估，但必须标注"题目缺失影响"。
7. 若指定轮次但未找到出题阶段（S1）文档，必须标注"轮次题目缺失影响"。
8. 不得以"等待补充后再评分"替代当前轮结论；必须先给出本轮评分，再附补充建议。
9. 微信输出必须为逐行结构文本，不得退化为自然段摘要。
10. 若结论为 `B/C` 不推荐、淘汰或不合格，`next_round_focus` 必须为空，且不得输出任何下一轮建议语句。
11. 评分证据必须来自候选人回答文本；面试官发言不得作为加减分依据。
12. 若结论为 `B/C`，必须输出 `eliminate_reasons` 数组，数组元素只能从以下原因中选择：技术能力/业务能力无法达到标准、能力经验与岗位不匹配、沟通能力较差、薪酬待遇不匹配、行业背景不匹配、学历不匹配、稳定性不强、理念风格与团队不符合、放弃面试、其他原因。

## 评分平衡与保守定级（强制）

1. 当输出中存在"加分项"时，必须尝试给出**数量对等**的"减分项/风险项"（1 个加分至少对应 1 个减分），避免结论单边乐观。
2. 若客观证据不足以给出对等减分项，必须明确标注 `risk_placeholder`（如"证据不足，按保守原则补记风险观察点"），不得直接省略。
3. `recommended_level` 必须采用保守定级原则：在"证据不完整、表现波动、分歧明显"任一情形下，默认下调半档到一档（例如 `P7` -> `P7-` 或 `P6+`）。
4. 只有在"稳定性 + 影响范围 + 复杂度处理"三项证据同时充分时，才允许给 `+` 档。
5. `grade` 与 `recommended_level` 冲突时，以更保守结果为准（宁可低估一档，不可高估一档）。
6. 若证据不足或冲突明显，必须在本轮内自动降档并明确标注"证据缺口与保守处理依据"，不得在评估阶段（S2）向用户发起补充证据交互。

## 与 workflow 协作

`interview-orchestrator` 会：

1. 收集题目、纪要、标准答案
2. 调用本 Skill
3. 输出评分总结后，立即向面试官发起反馈采集（强制询问）
4. 将"系统结论 + 面试官反馈"交给 `interview-memory` 做差异分析与学习沉淀
5. 调用 `interview-records.write_stage_document(stage=S2)` 落盘（评估阶段（S2））

## 面试官反馈采集（强制）

评估阶段（S2）总结输出后必须追加以下询问，不得省略：

1. `请给本次系统总结打分（1-10）`。
2. `你认同的面试评价等级是？（A+/A/B+/B/C）`。
3. `你认同的推荐职级是？（仅代码级：P6-...P8+ 或 不推荐）`。
4. `你认为系统结论需要修正的关键点是什么？`。

输出约束：

- 若面试官未即时回复，必须标注 `interviewer_feedback_status: pending`，并返回"待补反馈"提醒。
- 一旦收到反馈，必须生成 `system_vs_interviewer_analysis`，至少包含：一致点、分歧点、可执行学习动作。
- 状态推进规范：
  - 输出系统评分时写 `state=draft`。
  - 发起并等待面试官反馈时写 `state=collect_feedback`。
  - 生成差异分析并完成落盘时写 `state=finalize`。
