# Requirements Document

## Introduction

本规格旨在对 `apps/web` 前端进行结构化整改，解决当前存在的数据访问边界不清、状态流重复副作用、流式消息实现脆弱、错误处理不一致、HTML 渲染安全基线不统一以及页面职责过重等问题。

该特性不是新增业务功能，而是一次前端架构治理与能力补强。目标是在不改变核心业务流程的前提下，建立可持续演进的前端实现标准，使候选人管理、导入流程、LUI 工作台等核心模块具备更强的一致性、可靠性、安全性和可维护性。

## Alignment with Product Vision

该整改方案与 `product.md` 中的以下产品目标直接对齐：

- 支持“候选人管理、简历导入、AI 工作台、共享”等核心功能形成稳定闭环
- 保证主列表与工作台在本地优先架构下保持可维护和可回溯
- 避免导入流程、LUI 工作台等关键能力因前端结构混乱而影响交付效率与用户体验
- 为后续 1000+ 候选人列表浏览、非阻塞导入处理等产品目标提供可持续的前端基础

## Requirements

### Requirement 1

**User Story:** 作为技术面试官，我希望前端不同页面和模块遵循统一的数据访问与状态管理方式，以便在候选人管理、导入和 AI 工作台之间获得一致、可靠的操作体验。

#### Acceptance Criteria

1. WHEN 任一页面、组件或 store 需要访问后端接口 THEN 系统 SHALL 通过统一的数据访问层完成请求，而不是在页面或组件内直接使用原始 `fetch`
2. IF 请求为 JSON、FormData、文本或流式响应 THEN 系统 SHALL 通过协议感知的 API 客户端正确处理请求头、响应解析、错误映射与取消控制
3. WHEN 功能组件触发业务行为 THEN 系统 SHALL 通过单向数据流协调副作用，避免组件内部调用 store action 与向父层重复发出同语义事件
4. WHEN 页面卸载、路由切换或用户发起新的同类请求 THEN 系统 SHALL 正确取消或忽略过期请求，防止状态被旧结果覆盖

### Requirement 2

**User Story:** 作为使用 AI 工作台的面试官，我希望 LUI 的会话、消息、候选人绑定和文件资源操作稳定可靠，以便在工作台中持续完成面试辅助工作而不出现状态错乱。

#### Acceptance Criteria

1. WHEN 用户进入 LUI 页面 THEN 系统 SHALL 明确初始化会话列表与当前会话状态
2. WHEN 用户切换、创建或删除会话 THEN 系统 SHALL 只执行一次对应副作用，不得产生重复请求或竞态更新
3. WHEN 用户发送流式消息 THEN 系统 SHALL 通过稳定的流式协议解析机制处理文本片段、完成事件和错误事件
4. IF 流式消息中断、解析失败或网络异常 THEN 系统 SHALL 提供一致的失败状态、可恢复策略与用户反馈
5. WHEN 用户为会话绑定或解绑候选人 THEN 系统 SHALL 持久化该变更，而不是仅更新本地内存状态
6. WHEN 用户上传或删除文件资源 THEN 系统 SHALL 通过统一契约完成操作，并在页面中得到一致的状态反馈

### Requirement 3

**User Story:** 作为招聘协调同学，我希望导入页面具备清晰、稳定、可恢复的轮询与文件处理机制，以便高效跟踪批量导入任务而不出现 UI 抖动或重复请求。

#### Acceptance Criteria

1. WHEN 导入页面轮询批次状态 THEN 系统 SHALL 使用可控的串行轮询机制，避免异步重入
2. IF 当前不存在进行中的导入批次 THEN 系统 SHALL 停止或暂停轮询，避免无效请求
3. WHEN 用户重试失败文件、取消批次或查看文件详情 THEN 系统 SHALL 通过导入领域 API 模块完成请求，不得在页面中直接构造原始网络调用
4. WHEN 用户选择导入文件 THEN 系统 SHALL 通过统一的平台适配或文件选择能力完成，而不是在页面内散落 DOM 与平台特定逻辑
5. IF 导入过程发生冲突、失败或部分成功 THEN 系统 SHALL 提供一致、可理解的状态展示与交互反馈

### Requirement 4

**User Story:** 作为前端开发者，我希望错误处理、通知机制和 HTML 渲染安全策略统一，以便在维护候选人页面、导入页面和 LUI 页面时降低隐患并提高调试效率。

#### Acceptance Criteria

1. WHEN 任意模块发生用户可感知错误 THEN 系统 SHALL 通过统一的通知机制向用户反馈，而不是混用 `alert`、静默失败和零散日志
2. IF 错误属于网络失败、服务端错误、业务错误或用户输入错误 THEN 系统 SHALL 使用统一错误模型进行归类和处理
3. WHEN 页面需要渲染 Markdown 或 HTML 内容 THEN 系统 SHALL 通过统一的安全渲染器生成内容，不得在不同组件中使用不一致的净化策略
4. IF 某段错误被有意忽略 THEN 系统 SHALL 明确说明忽略原因并保留必要的调试信息，不得使用无上下文的空 `catch` 或 `.catch(() => undefined)`

### Requirement 5

**User Story:** 作为前端开发者，我希望候选人列表页、导入页和 LUI 页的职责边界清晰、文件规模可控，以便后续新增功能时能够低风险演进。

#### Acceptance Criteria

1. WHEN 页面承担多个独立职责（如布局、数据请求、轮询、文件选择、弹窗流程、格式化逻辑） THEN 系统 SHALL 将其拆分为更聚焦的组件、composable 或领域模块
2. WHEN 页面或 store 文件持续增长 THEN 系统 SHALL 按领域职责拆分，而不是继续堆叠逻辑到单文件中
3. WHEN 功能逻辑可在多个页面复用 THEN 系统 SHALL 抽取为可复用的 composable、格式化器或布局组件
4. IF 某个组件或模块已无有效引用 THEN 系统 SHALL 将其识别并清理，避免死代码长期存在

### Requirement 6

**User Story:** 作为维护该项目的团队成员，我希望前端具备明确的类型约束、测试覆盖和开发规则，以便减少回归风险并阻止旧问题再次出现。

#### Acceptance Criteria

1. WHEN 开发者引入新的平台适配字段或复杂响应结构 THEN 系统 SHALL 通过显式类型、共享 DTO 或适配器函数建模，而不是使用宽泛断言绕过类型系统
2. WHEN 新增或修改数据访问、流式解析、轮询逻辑或关键组件交互 THEN 系统 SHALL 具备相应的单元测试或行为测试覆盖
3. WHEN 代码进入合并流程 THEN 系统 SHALL 能通过类型检查，并满足前端治理规则（如禁止页面/组件直接 `fetch`、禁止 `alert`、禁止空 `catch`、禁止危险断言）
4. IF 团队新增页面或功能模块 THEN 系统 SHALL 有可遵循的前端实现规范，延续本次整改建立的分层与治理标准

## Non-Functional Requirements

### Code Architecture and Modularity
- **单一职责**：页面、组件、store、API、composable 各自只承担清晰职责
- **模块化设计**：LUI、Import、Candidates 等核心域需按领域拆分
- **依赖控制**：组件不得跨层直接依赖协议细节或平台细节
- **清晰接口**：父子组件事件契约、领域 API、错误模型、渲染器都必须显式定义

### Performance
- 主列表搜索交互应支持取消与防抖，避免旧请求覆盖新结果
- 导入批次轮询应避免异步重入与无效轮询
- LUI 消息流更新应支持增量渲染且避免重复状态变更
- 页面拆分后不得引入明显性能回退

### Security
- 所有 HTML/Markdown 渲染必须经过统一安全净化
- 文件上传与表单提交需正确处理协议头，避免错误配置导致的潜在风险
- 前端不得通过分散实现削弱鉴权、错误处理或接口安全边界

### Reliability
- 对网络错误、服务端错误、流式中断、轮询失败有一致处理策略
- 关键异步行为支持取消、重试或退避
- 不允许重复副作用、静默吞错和本地假持久化

### Usability
- 用户应获得一致的成功、失败、加载中反馈
- LUI、导入、候选人页面的交互行为应符合直觉，避免“看起来成功但未真正生效”
- 整改不应破坏现有核心业务路径的可用性
